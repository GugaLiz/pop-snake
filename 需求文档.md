# 消消蛇需求文档

## 1. 项目概述

### 1.1 项目名称
消消蛇

### 1.2 项目定位
一款融合「贪吃蛇走位压力」与「三消构筑策略」的轻量休闲小游戏。

### 1.3 核心卖点
- 玩家既要像贪吃蛇一样规避碰撞、规划路线，又要像消消乐一样安排颜色顺序。
- 吃色块会增长，连续同色达到条件后会消除，形成「变长带来风险，消除带来喘息」的动态循环。
- 玩法门槛低，但有明显策略空间和连击爽感，适合做短局、高复玩体验。

### 1.4 目标玩家
- 喜欢贪吃蛇、消除、肉鸽轻策略类休闲游戏的玩家
- 偏好 1 到 5 分钟短局体验的移动端或网页端玩家

## 2. 设计目标

### 2.1 核心目标
- 保留贪吃蛇的即时操作乐趣
- 引入颜色构筑与消除决策，提升耐玩性
- 通过消除机制抑制蛇无限增长，避免玩法过早失控
- 用少量规则做出较强的“再来一局”驱动力

### 2.2 体验关键词
- 紧张
- 策略
- 爽快
- 短平快
- 可复盘

## 3. 核心玩法说明

### 3.1 基础循环
1. 玩家控制蛇在网格中移动。
2. 地图中随机散落不同颜色的色块。
3. 蛇头吃到色块后，蛇身长度增加 1 节，新增长的节带有对应颜色。
4. 当蛇身上出现连续 3 节及以上的同色段时，触发消除。
5. 被消除的蛇身片段从身体中移除，蛇长度缩短，玩家获得分数与短暂的场面缓冲。
6. 玩家在“继续增长”与“主动制造消除”之间循环博弈，尽量存活更久并获得更高分数。

### 3.2 玩法本质
本作不是传统意义上“吃得越多越好”的贪吃蛇，而是“控制长度、构筑颜色、制造消除”的生存策略游戏。

### 3.3 失败条件
满足任一条件则本局结束：
- 蛇头撞到墙壁
- 蛇头撞到自身
- 特殊模式下达成额外失败条件，例如倒计时归零且未完成目标

### 3.4 胜利/结算方式
- 无尽模式：以分数、生存时长、总消除数作为结算指标
- 关卡模式：满足当前关卡目标则过关

## 4. 核心规则设计

### 4.1 蛇身与颜色规则
- 蛇初始长度为 3 节
- 每节蛇身都具备颜色属性
- 蛇头吃到的色块不会立即在头部生效，而是作为新节加入蛇尾
- 蛇的颜色序列因此会随着吃块顺序不断变化

设计原因：
- 让“吃到什么”和“最终如何排列到蛇身上”形成延迟反馈
- 提高玩家对整体蛇身结构的观察与规划需求

### 4.2 消除触发规则
- 当蛇身出现连续 3 节及以上同色时，立即触发消除
- 默认仅消除该连续区段，不连带相邻异色区段
- 单次可消除长度大于 3，例如连续 4 节或 5 节同色时全部消除
- 消除完成后，前后剩余蛇身自动重新连接

### 4.3 连锁判定规则
- 一次消除后，如果新拼接出的蛇身再次形成连续 3 节及以上同色，则触发追加消除
- 该情况计为连锁消除，可触发额外分数加成和表现强化

### 4.4 颜色数量
- MVP 阶段使用 4 种基础颜色
- 后续版本可扩展到 5 种
- 不建议超过 5 种，避免记忆负担过高、策略感反而下降

## 5. 地图与生成规则

### 5.1 棋盘规格
- 推荐默认棋盘为 12x12 或 15x15
- MVP 优先使用 12x12，便于压缩单局时长与调试节奏

### 5.2 色块生成原则
- 地图上需持续存在一定数量的可吃色块
- 色块不应完全无脑随机，应保证基础颜色分布相对均衡
- 避免连续过长时间刷不出玩家急需的颜色，导致局面纯靠运气

### 5.3 推荐生成机制
- 地图维持一个目标色块数量，例如 10 到 16 个
- 每次有色块被吃掉或地图空位增多时，按规则补充
- 补充时优先从当前全局数量较少的颜色池中抽取，降低极端失衡概率

### 5.4 无解保护
- 系统需避免长时间出现“地图上没有任何可能帮助形成三消的颜色结构”
- 可通过以下方式兜底：
- 若连续一定时间未发生消除，则提高蛇身已有高频颜色的刷新权重
- 若地图颜色分布过散，则触发一次平衡刷新

## 6. 分数与反馈系统

### 6.1 基础得分
- 吃到色块：获得少量基础分
- 消除蛇身：按消除节数获得主要分数
- 连锁消除：额外加成

### 6.2 Combo 机制
- 2 秒内连续发生消除时，Combo 层数提升
- 每层 Combo 提供分数倍率提升
- 建议倍率从 `1.0` 开始，每层 `+0.2`，上限 `2.0`

### 6.3 反馈表现
- 吃块时：轻提示音、色块吸附到蛇尾的表现
- 消除时：清脆音效、蛇身闪白、粒子散落、短暂停顿
- 连锁时：屏幕文字提示，如 `Combo x2`、`Combo x3`
- 高危长度时：通过 UI 或蛇身描边提醒玩家当前风险升高

## 7. 节奏与难度设计

### 7.1 核心矛盾
- 蛇越长，移动风险越高
- 但只有主动吃块，才有机会完成消除
- 玩家需要不断平衡“冒险增长”和“及时减负”

### 7.2 难度曲线建议
- 前期：色块较密集，便于理解规则和制造第一次消除
- 中期：开始要求玩家绕路、预判、为特定颜色做准备
- 后期：空地增多、可用色块变稀，走位压力提升

### 7.3 长度压力机制
- 当蛇长度超过初始长度的 3 倍后，可略微提升移动速度
- 该机制用于制造后期压力，避免玩家靠保守拖局

## 8. 游戏模式设计

### 8.1 无尽模式
定位：
最适合作为首发模式，验证核心玩法是否成立。

规则：
- 无固定终点
- 目标是追求更高分数、更长生存时间、更高连击

### 8.2 限时模式
规则：
- 在 60 秒内完成指定数量的消除或达到指定分数

体验目标：
- 提升操作压迫感，强化爽感而非深度运营

### 8.3 限步模式
规则：
- 在限定步数内完成目标，例如达成指定消除数或最终长度要求

体验目标：
- 强化规划感，弱化纯反应要求

### 8.4 精准模式
规则：
- 结束时蛇长度必须刚好达到目标值

体验目标：
- 鼓励玩家在“吃”和“消”之间精确控制

## 9. 特殊色块设计

### 9.1 设计原则
- 特殊色块应服务于核心玩法，而不是喧宾夺主
- 首发版本不宜过多，避免学习成本上升

### 9.2 炸弹色块
效果：
- 吃掉后直接消除蛇尾 3 节

用途：
- 应急减压
- 打破过长蛇身带来的死局

### 9.3 彩虹色块
效果：
- 可视为任意颜色参与连续同色判定

用途：
- 提高构筑弹性
- 帮助玩家达成关键消除

### 9.4 减速色块
效果：
- 蛇移动速度降低 50%，持续 5 秒

用途：
- 给予新手缓冲
- 在高压阶段提供操作修正空间

### 9.5 投放建议
- 特殊色块在第二版加入
- 地图上同一时间特殊色块数量应严格受控
- 彩虹色块应保持稀有，避免破坏颜色策略的价值

## 10. UI 与表现需求

### 10.1 整体视觉基调
整体风格参考提供的配图，视觉方向定义为：
- 明亮、活泼、街机感强
- 高饱和暖色背景配合清晰深色描边
- 图形偏圆润、偏贴纸感，而不是写实或极简科技风
- 第一眼要传达“轻松、好玩、上头”，而不是复杂或硬核

推荐主色氛围：
- 主背景色：橙黄色系，营造温暖、醒目的主视觉氛围
- 强调色：亮黄、黄绿、青绿，用于高亮色块、分数反馈、按钮强调
- 中和色：深棕、栗色，用于辅助图形、装饰物、关卡主题元素
- 轮廓色：接近纯黑的深色描边，用于统一图标、蛇身、按钮和色块的边缘表现

风格要求：
- 所有核心元素尽量有明确描边，增强识别度
- 避免灰、白、半透明玻璃质感等偏弱存在感的设计
- 避免写实材质、金属质感、赛博霓虹等不相关风格

### 10.2 主界面信息
- 当前分数
- 当前长度
- 当前 Combo
- 模式目标或剩余时间
- 暂停按钮

### 10.3 棋盘信息表达
- 普通色块使用高辨识度颜色与统一形状
- 特殊色块增加花纹、边框或图标，避免仅靠颜色区分
- 蛇身每节颜色必须清晰可见
- 色块与蛇身推荐采用圆角方块、圆润胶囊块或贴纸块风格
- 元素边缘建议使用较粗描边，保证小尺寸下仍清晰

### 10.4 色彩应用建议
- 棋盘背景不宜过花，优先使用干净的大色块底或轻纹理渐变底
- 普通色块颜色应与整体主视觉统一，避免出现过冷或过灰的突兀颜色
- 危险状态可用红橙色闪烁提示，但整体仍要服从暖色主基调
- Combo、消除特效优先使用亮黄、浅绿、白色闪光，形成“啪一下弹开”的爽感

### 10.5 辅助提示
- 高亮“还差 1 节即可触发消除”的同色段
- 可选地对下一节将接入蛇尾的位置做弱提示
- 新手引导中展示 1 次完整的“吃块 -> 增长 -> 三消”流程

### 10.6 动效要求
- 吃块反馈应短促明确，不拖慢节奏
- 消除动画时长要克制，建议 150ms 到 250ms
- 连锁触发时可适度增加停顿与镜头震动，但不能影响操作手感
- 动效曲线偏弹性、轻快，符合街机贴纸风格
- UI 弹出提示尽量采用“跳出”“弹开”“闪一下”的反馈，而不是柔和淡入淡出

## 11. 新手引导需求

### 11.1 首局引导目标
- 让玩家理解颜色会进入蛇身
- 让玩家理解连续 3 节同色会消除
- 让玩家理解“控制长度”比“无限变长”更重要

### 11.2 引导方式
- 采用弱打断式引导
- 前 30 秒内分步展示 2 到 3 条提示
- 首局可适当提高目标颜色刷新概率，帮助玩家尽快打出第一次消除

## 14. 技术实现建议

### 14.1 前后端技术栈建议

#### 前端技术栈
- 核心语言：TypeScript
- 构建工具：Vite
- UI 框架：React
- 游戏渲染：Phaser 3
- 样式方案：CSS Modules 或 SCSS
- 状态管理：首版优先使用 React 内状态 + 模块化游戏状态管理，避免过早引入复杂全局状态库

推荐原因：
- TypeScript 适合管理蛇身、色块、关卡配置等结构化数据，能减少规则类 Bug
- Vite 启动快、构建轻，适合独立小游戏项目快速迭代
- React 适合承载首页、结算页、HUD、模式选择、设置面板等界面层
- Phaser 3 适合 2D 网格、精灵、输入控制、动画和音效管理，比纯 Canvas 手写更省开发成本
- React + Phaser 的组合适合把“游戏内逻辑”和“外围界面系统”分层处理

前端分层建议：
- React 负责页面框架、菜单、结算、引导、设置、排行榜等非核心实时画面
- Phaser 负责主战斗场景、蛇移动、色块生成、碰撞判定、消除动画和局内特效

#### 后端技术栈
- 运行环境：Node.js
- 服务框架：NestJS 或 Express
- 数据库与后端服务：Supabase
- 接口协议：REST API
- 鉴权方式：优先使用 Supabase Auth，首版也可先采用匿名用户或弱登录方案

推荐原因：
- 这个项目首版以后端弱依赖为主，主要需求集中在排行榜、存档、玩家配置、活动配置等轻服务能力
- Node.js 与前端 TypeScript 技术栈统一，开发和维护成本更低
- 如果希望结构更规范、方便后续扩展活动/关卡/用户系统，优先选 NestJS
- 如果首版只做极轻量排行榜和配置下发，Express 也足够
- Supabase 基于 PostgreSQL，天然适合排行榜、玩家数据、配置表和简单后台能力
- Supabase 自带 Auth、数据库、存储和行级权限能力，能显著降低独立小游戏后端搭建成本

后端职责建议：
- 用户基础信息管理
- 分数上传与排行榜
- 关卡配置或活动配置下发
- 玩家设置、引导进度、解锁状态存储

#### 部署建议
- 前端：Vercel、Netlify 或静态资源 CDN 部署
- 后端：Node 服务部署到云服务器、Railway、Render 或国内云平台；轻量场景下也可优先直接依赖 Supabase 能力，减少自建接口
- 数据层：Supabase 托管 PostgreSQL，统一承载用户、成绩、存档与配置数据

#### MVP 技术栈建议结论
若目标是尽快做出可玩版本，推荐采用以下组合：
- 前端：TypeScript + Vite + React + Phaser 3
- 数据与服务：Supabase
- 后端：可暂时不单独开发业务服务，先由前端直连 Supabase 完成排行榜、存档和基础配置读取；如后续逻辑变复杂，再增加 Node.js 服务层

这样可以先把核心玩法跑通，把主要精力放在手感、节奏和消除规则打磨上。

### 14.2 核心数据结构
- 蛇身建议使用有序数组或双端队列维护
- 每节至少记录：网格坐标、颜色、节序
- 消除判定可基于颜色序列扫描连续段完成

### 14.3 工程目录建议
- `src/app`：React 页面层，如首页、模式选择、结算页、设置页
- `src/game`：Phaser 游戏核心逻辑
- `src/game/scenes`：场景，如加载、主游戏、结算过场
- `src/game/entities`：蛇、色块、特效等实体
- `src/game/systems`：碰撞、生成、消除、计分、关卡规则等系统
- `src/config`：颜色、数值、模式、关卡参数配置
- `src/assets`：图片、音效、字体资源
- `server`：后端服务目录，可在后续版本接入

### 14.4 关键技术难点
- 吃块后新增蛇节与移动逻辑的同步
- 中间片段消除后，蛇身坐标如何重新衔接
- 连锁消除的递归或循环判定
- 色块生成的平衡性与无解保护

### 14.5 推荐实现顺序
1. 完成基础蛇移动与碰撞
2. 实现普通色块生成与吃块增长
3. 实现颜色序列判定与消除
4. 实现分数、反馈和连锁
5. 最后追加特殊色块和模式扩展

## 15. Supabase 数据设计草案

### 15.1 设计目标
- 以最小数据模型支撑 MVP：登录、排行榜、基础存档、配置读取
- 控制表数量，优先满足首版可用，避免过度建模
- 让前端可在早期直接接入 Supabase，后续再平滑增加服务层

### 15.2 推荐表结构

#### `profiles`
用途：
- 存储玩家基础资料

建议字段：
- `id`：UUID，主键，对应 `auth.users.id`
- `nickname`：文本，玩家昵称
- `avatar_url`：文本，头像地址，可为空
- `created_at`：时间
- `updated_at`：时间

说明：
- 每个登录用户一条记录
- 首版即使使用匿名登录，也建议保留该表，便于后续升级账号体系

#### `player_settings`
用途：
- 存储玩家本地偏好和通用设置

建议字段：
- `id`：UUID，主键
- `user_id`：UUID，关联 `profiles.id`
- `bgm_enabled`：布尔
- `sfx_enabled`：布尔
- `vibration_enabled`：布尔
- `language`：文本
- `updated_at`：时间

说明：
- 若设置项不多，也可直接并入 `profiles`
- 若后续设置项扩展较多，独立成表更清晰

#### `run_records`
用途：
- 存储每一局对战记录，用于排行榜、战绩和数据分析

建议字段：
- `id`：UUID，主键
- `user_id`：UUID，关联 `profiles.id`，匿名用户可为空或使用匿名 id
- `mode`：文本，如 `endless`、`timed`、`steps`
- `score`：整数
- `survival_seconds`：整数
- `final_length`：整数
- `max_combo`：整数
- `total_eliminated`：整数
- `seed`：文本或整数，可选，用于复盘或问题排查
- `client_version`：文本
- `created_at`：时间

说明：
- 这是首版最核心的数据表
- 排行榜、个人历史、局均表现都可基于该表计算

#### `player_progress`
用途：
- 存储玩家解锁与引导进度

建议字段：
- `id`：UUID，主键
- `user_id`：UUID，关联 `profiles.id`
- `tutorial_completed`：布尔
- `highest_score`：整数
- `highest_combo`：整数
- `best_survival_seconds`：整数
- `unlocked_modes`：JSONB 或文本数组
- `updated_at`：时间

说明：
- 若后续加入成就、任务、局外成长，可继续扩展该表或拆表

#### `game_configs`
用途：
- 存储可热更新的游戏配置

建议字段：
- `id`：UUID，主键
- `config_key`：文本，唯一，例如 `core_balance_v1`
- `config_value`：JSONB
- `is_active`：布尔
- `updated_at`：时间

说明：
- 用于存储颜色数量、基础速度、地图尺寸、特殊色块权重等参数
- 前端启动时可拉取当前激活配置，实现轻量热更新

### 15.3 MVP 最小表集合
如果目标是最快上线，首版只需要这 4 张表：
- `profiles`
- `run_records`
- `player_progress`
- `game_configs`

`player_settings` 可视开发节奏延后。

### 15.4 排行榜实现建议
- 全局排行榜基于 `run_records` 按模式和分数排序
- 个人最佳成绩可通过视图、物化视图或前端聚合获取
- 若后续数据量变大，可增加独立排行榜表或周期性聚合任务

推荐排行榜维度：
- 无尽模式最高分
- 单局最长生存时间
- 单局最高 Combo
- 单局最多消除数

### 15.5 Supabase Auth 接入建议
- MVP 可优先使用匿名登录，减少注册门槛
- 若需要跨设备同步，再引导用户绑定邮箱或第三方登录
- 所有用户数据表都应以 `user_id` 关联 `auth.users.id`

### 15.6 Row Level Security 建议
- `profiles`：用户仅可读写自己的资料
- `player_settings`：用户仅可读写自己的设置
- `player_progress`：用户仅可读写自己的进度
- `run_records`：用户可写入自己的战绩；排行榜场景可只开放经过筛选的只读查询
- `game_configs`：普通用户只读，后台管理员可写

### 15.7 接口访问建议
- 首版前端可直接通过 Supabase Client 访问 `profiles`、`player_progress`、`run_records`
- 涉及反作弊、复杂结算校验、活动结算时，再增加 Node.js 服务层或 Supabase Edge Functions
- 分数提交若担心作弊，后续可将“结算入库”改为服务端校验后写入

### 15.8 未来扩展表
如项目验证成功，可追加以下表：
- `achievements`：成就定义
- `player_achievements`：玩家成就进度
- `events`：限时活动
- `skins`：皮肤定义
- `player_skins`：玩家皮肤解锁情况
- `daily_tasks`：每日任务
- `replay_records`：回放数据或关键帧记录

### 15.9 数据设计结论
`Supabase` 适合这个项目的关键原因在于：
- 首版可以少写很多后端基础设施
- 数据模型天然适配排行榜、存档和配置下发
- 后续可逐步补上鉴权、函数、存储和运营能力

建议开发策略是：
- 先用最少表结构把核心玩法跑通
- 先做“能记录、能排行、能同步基础进度”
- 再根据真实玩法迭代补充成就、活动和局外成长

## 16. 验收标准

### 16.1 核心玩法验收
- 玩家能在 1 局内自然理解“增长”和“消除”的关系
- 游戏不会频繁出现纯靠运气、无从决策的局面
- 三消成功时有明显正反馈
- 玩家可以通过策略控制长度，而非只能被动等死

### 16.2 MVP 完成标准
- 单局流程完整可玩
- 核心循环稳定，无明显逻辑错误
- 消除、连锁、结算逻辑正确
- 平均单局时长控制在 1 到 3 分钟

## 17. 风险与注意事项

### 17.1 主要风险
- 如果消除太容易，游戏会失去贪吃蛇的紧张感
- 如果消除太难，玩家会觉得只是“更折磨的贪吃蛇”
- 如果色块刷新过于随机，策略空间会被运气吞没
- 如果动画太重，会破坏移动手感

### 17.2 设计把控建议
- 优先把核心循环打磨顺，再扩玩法
- 先验证“连续同色消除”是否真的有趣，再决定要不要上复杂系统
- 特殊色块、Boss、养成都应建立在基础体验稳定之后

## 18. 结论
“消消蛇”的产品核心不是简单叠加两个经典玩法，而是围绕“颜色进入蛇身后形成可规划的消除链”建立新的生存循环。

首版最重要的不是功能多，而是把以下三个感受做出来：
- 吃块会紧张，因为会变长
- 凑消除会开心，因为能减负
- 每一步都值得想，因为颜色顺序会影响后续生死

如果这三个感受成立，这个项目就具备继续扩展关卡、特殊机制和长期成长系统的价值。
