# 消消蛇需求文档

## 0. 当前开发范围说明

### 0.1 当前阶段目标
当前优先开发一版“不需要后端接口”的纯前端网页版本，目标是在 PC 和移动端浏览器中都能完整体验核心玩法。

本阶段重点是验证核心循环是否成立：移动、吃色块、增长、同色三消、连锁、计分、失败、重新开始。

### 0.2 当前阶段不接入的能力
以下能力暂不作为当前版本开发内容，只作为后续扩展预留：
- 后端业务接口
- Supabase 或其他云数据库
- 登录、账号体系、跨设备同步
- 在线排行榜
- 服务端存档
- 服务端配置下发
- 反作弊校验

### 0.3 当前阶段数据策略
- 游戏配置、数值参数先写在前端配置文件中。
- 最高分、设置项、引导完成状态等轻量数据使用浏览器 `localStorage` 保存。
- 单局过程数据只保存在内存中，刷新页面后不保留。
- 后续如果需要排行榜、跨设备同步或活动配置，再接入 Supabase / Node.js 服务层。

### 0.4 适配要求
- PC 端支持键盘方向键和 `WASD` 操作。
- 移动端支持滑动方向操作，并提供可选虚拟方向键。
- 页面需要适配桌面、平板、手机竖屏，不要求首版支持复杂横屏布局。
- 棋盘需要根据屏幕尺寸自适应，保证核心可玩区域完整可见。

## 1. 项目概述

### 1.1 项目名称
消消蛇

### 1.2 项目定位
一款融合「贪吃蛇走位压力」与「三消构筑策略」的轻量休闲小游戏。

### 1.3 核心卖点
- 玩家既要像贪吃蛇一样规避碰撞、规划路线，又要像消消乐一样安排颜色顺序。
- 吃色块会增长，连续同色达到条件后会消除，形成「变长带来风险，消除带来喘息」的动态循环。
- 玩法门槛低，但有明显策略空间和连击爽感，适合做短局、高复玩体验。

### 1.4 目标玩家
- 喜欢贪吃蛇、消除、肉鸽轻策略类休闲游戏的玩家
- 偏好 1 到 5 分钟短局体验的移动端或网页端玩家

## 2. 设计目标

### 2.1 核心目标
- 保留贪吃蛇的即时操作乐趣
- 引入颜色构筑与消除决策，提升耐玩性
- 通过消除机制抑制蛇无限增长，避免玩法过早失控
- 用少量规则做出较强的“再来一局”驱动力

### 2.2 体验关键词
- 紧张
- 策略
- 爽快
- 短平快
- 可复盘

## 3. 核心玩法说明

### 3.1 基础循环
1. 玩家控制蛇在网格中移动。
2. 地图中随机散落不同颜色的色块。
3. 蛇头吃到色块后，蛇身长度增加 1 节，新增长的节带有对应颜色。
4. 每次吃到色块后，色块颜色作为新节加入蛇尾；当蛇尾最后 3 节颜色完全相同时，触发消除。
5. 被消除的是蛇尾最后 3 节同色片段，蛇长度缩短，玩家获得分数与短暂的场面缓冲。
6. 玩家通过规划吃色块顺序，让蛇尾末端形成同色三连，在“继续增长”与“主动制造消除”之间循环博弈。

### 3.2 玩法本质
本作不是传统意义上“吃得越多越好”的贪吃蛇，而是“控制长度、构筑颜色、制造消除”的生存策略游戏。

### 3.3 失败条件
满足任一条件则本局结束：
- 蛇头撞到墙壁
- 蛇头撞到自身
- 特殊模式下达成额外失败条件，例如倒计时归零且未完成目标

### 3.4 胜利/结算方式
- 无尽模式：以分数、生存时长、总消除数作为结算指标
- 关卡模式：满足当前关卡目标则过关

## 4. 核心规则设计

### 4.1 蛇身与颜色规则
- 蛇初始长度为 3 节
- 每节蛇身都具备颜色属性
- 蛇头吃到的色块不会立即在头部生效，而是作为新节加入蛇尾
- 蛇的颜色序列因此会随着吃块顺序不断变化

设计原因：
- 让“吃到什么”和“最终如何排列到蛇身上”形成延迟反馈
- 提高玩家对整体蛇身结构的观察与规划需求

### 4.2 消除触发规则
- 每次吃到色块后，先将该色块颜色作为新蛇节加入蛇尾
- 系统只检查蛇尾最后 3 节颜色是否完全一致
- 若蛇尾最后 3 节同色，则立即消除这 3 节尾巴
- 若最后 3 节不完全同色，则不消除，颜色继续保留在蛇尾序列中

### 4.3 连锁判定规则
- 当前首版不做蛇身中段连锁消除，只以每次吃块后蛇尾末端三连作为触发条件
- 若短时间内多次触发尾巴三连消除，则计为 Combo，可触发额外分数加成和表现强化

### 4.4 颜色数量
- MVP 阶段使用 4 种基础颜色
- 后续版本可扩展到 5 种
- 不建议超过 5 种，避免记忆负担过高、策略感反而下降

## 5. 地图与生成规则

### 5.1 棋盘规格
- 推荐默认棋盘为 12x12 或 15x15
- MVP 优先使用 12x12，便于压缩单局时长与调试节奏

### 5.2 色块生成原则
- 地图上需持续存在一定数量的可吃色块
- 色块不应完全无脑随机，应保证基础颜色分布相对均衡
- 避免连续过长时间刷不出玩家急需的颜色，导致局面纯靠运气

### 5.3 推荐生成机制
- 地图维持一个目标色块数量，例如 10 到 16 个
- 每次有色块被吃掉或地图空位增多时，按规则补充
- 补充时优先从当前全局数量较少的颜色池中抽取，降低极端失衡概率

### 5.4 无解保护
- 系统需避免长时间出现“地图上没有任何可能帮助形成三消的颜色结构”
- 可通过以下方式兜底：
- 若连续一定时间未发生消除，则提高蛇身已有高频颜色的刷新权重
- 若地图颜色分布过散，则触发一次平衡刷新

## 6. 分数与反馈系统

### 6.1 基础得分
- 吃到色块：获得少量基础分
- 消除蛇身：按消除节数获得主要分数
- 连锁消除：额外加成

### 6.2 Combo 机制
- 2 秒内连续发生消除时，Combo 层数提升
- 每层 Combo 提供分数倍率提升
- 建议倍率从 `1.0` 开始，每层 `+0.2`，上限 `2.0`

### 6.3 反馈表现
- 吃块时：轻提示音、色块吸附到蛇尾的表现
- 消除时：清脆音效、蛇身闪白、粒子散落、短暂停顿
- 连锁时：屏幕文字提示，如 `Combo x2`、`Combo x3`
- 高危长度时：通过 UI 或蛇身描边提醒玩家当前风险升高

## 7. 节奏与难度设计

### 7.1 核心矛盾
- 蛇越长，移动风险越高
- 但只有主动吃块，才有机会完成消除
- 玩家需要不断平衡“冒险增长”和“及时减负”

### 7.2 难度曲线建议
- 前期：色块较密集，便于理解规则和制造第一次消除
- 中期：开始要求玩家绕路、预判、为特定颜色做准备
- 后期：空地增多、可用色块变稀，走位压力提升

### 7.3 长度压力机制
- 当蛇长度超过初始长度的 3 倍后，可略微提升移动速度
- 该机制用于制造后期压力，避免玩家靠保守拖局

## 8. 游戏模式设计

当前纯前端版本已实现以下 5 种模式的基础版：标准、无尽、限时、限步、精准。标准模式为默认模式。

### 8.1 标准模式
定位：
默认模式，有边界限制，适合作为关卡式基础体验。

规则：
- 蛇头撞墙或撞到自身都会失败
- 达到当前关卡目标分数即可过关

### 8.2 无尽模式
定位：
最适合作为首发模式，验证核心玩法是否成立。

规则：
- 无固定终点
- 目标是追求更高分数、更长生存时间、更高连击

### 8.3 限时模式
规则：
- 在 60 秒内完成指定数量的消除或达到指定分数

体验目标：
- 提升操作压迫感，强化爽感而非深度运营

### 8.4 限步模式
规则：
- 在限定步数内完成目标，例如达成指定消除数或最终长度要求

体验目标：
- 强化规划感，弱化纯反应要求

### 8.5 精准模式
规则：
- 结束时蛇长度必须刚好达到目标值

体验目标：
- 鼓励玩家在“吃”和“消”之间精确控制

## 9. 特殊色块设计

当前纯前端版本已实现以下 3 种特殊色块的基础版：炸弹、彩虹、减速。

### 9.1 设计原则
- 特殊色块应服务于核心玩法，而不是喧宾夺主
- 首发版本不宜过多，避免学习成本上升

### 9.2 炸弹色块
效果：
- 吃掉后直接消除蛇尾 3 节

用途：
- 应急减压
- 打破过长蛇身带来的死局

### 9.3 彩虹色块
效果：
- 可视为任意颜色参与连续同色判定

用途：
- 提高构筑弹性
- 帮助玩家达成关键消除

### 9.4 减速色块
效果：
- 蛇移动速度降低 50%，持续 5 秒

用途：
- 给予新手缓冲
- 在高压阶段提供操作修正空间

### 9.5 投放建议
- 特殊色块基础版已加入当前纯前端版本，后续继续打磨投放权重、图标和动画表现
- 地图上同一时间特殊色块数量应严格受控
- 彩虹色块应保持稀有，避免破坏颜色策略的价值

## 10. UI 与表现需求

### 10.1 整体视觉基调
整体风格参考提供的配图，视觉方向定义为：
- 明亮、活泼、街机感强
- 高饱和暖色背景配合清晰深色描边
- 图形偏圆润、偏贴纸感，而不是写实或极简科技风
- 第一眼要传达“轻松、好玩、上头”，而不是复杂或硬核

推荐主色氛围：
- 主背景色：橙黄色系，营造温暖、醒目的主视觉氛围
- 强调色：亮黄、黄绿、青绿，用于高亮色块、分数反馈、按钮强调
- 中和色：深棕、栗色，用于辅助图形、装饰物、关卡主题元素
- 轮廓色：接近纯黑的深色描边，用于统一图标、蛇身、按钮和色块的边缘表现

风格要求：
- 所有核心元素尽量有明确描边，增强识别度
- 避免灰、白、半透明玻璃质感等偏弱存在感的设计
- 避免写实材质、金属质感、赛博霓虹等不相关风格

### 10.2 主界面信息
- 当前分数
- 当前长度
- 当前 Combo
- 模式目标或剩余时间
- 暂停按钮

### 10.3 棋盘信息表达
- 普通色块使用高辨识度颜色与统一形状
- 特殊色块增加花纹、边框或图标，避免仅靠颜色区分
- 蛇身每节颜色必须清晰可见
- 色块与蛇身推荐采用圆角方块、圆润胶囊块或贴纸块风格
- 元素边缘建议使用较粗描边，保证小尺寸下仍清晰

### 10.4 色彩应用建议
- 棋盘背景不宜过花，优先使用干净的大色块底或轻纹理渐变底
- 普通色块颜色应与整体主视觉统一，避免出现过冷或过灰的突兀颜色
- 危险状态可用红橙色闪烁提示，但整体仍要服从暖色主基调
- Combo、消除特效优先使用亮黄、浅绿、白色闪光，形成“啪一下弹开”的爽感

### 10.5 辅助提示
- 高亮“还差 1 节即可触发消除”的同色段
- 可选地对下一节将接入蛇尾的位置做弱提示
- 新手引导中展示 1 次完整的“吃块 -> 增长 -> 三消”流程

### 10.6 动效要求
- 吃块反馈应短促明确，不拖慢节奏
- 消除动画时长要克制，建议 150ms 到 250ms
- 连锁触发时可适度增加停顿与镜头震动，但不能影响操作手感
- 动效曲线偏弹性、轻快，符合街机贴纸风格
- UI 弹出提示尽量采用“跳出”“弹开”“闪一下”的反馈，而不是柔和淡入淡出

## 11. 新手引导需求

### 11.1 首局引导目标
- 让玩家理解颜色会进入蛇身
- 让玩家理解连续 3 节同色会消除
- 让玩家理解“控制长度”比“无限变长”更重要

### 11.2 引导方式
- 采用弱打断式引导
- 前 30 秒内分步展示 2 到 3 条提示
- 首局可适当提高目标颜色刷新概率，帮助玩家尽快打出第一次消除

## 12. 数值建议

### 12.1 MVP 默认参数
- 初始长度：3
- 基础颜色数：4
- 棋盘大小：12x12
- 地图常驻色块数：12
- 消除阈值：蛇尾最后连续 3 节同色
- Combo 判定窗口：2 秒

### 12.2 后续可调参数
- 移动速度
- 特殊色块刷新概率
- 地图常驻色块数
- 颜色数量
- 长度压力触发阈值
- 不同模式目标值

## 14. 技术实现建议

### 14.1 当前版本技术栈
当前版本只开发纯前端网页，不依赖任何后端接口。

推荐技术栈：
- 核心语言：TypeScript
- 构建工具：Vite
- UI 框架：React
- 游戏渲染：Phaser 3
- 样式方案：CSS Modules 或 SCSS，也可根据项目节奏使用普通 CSS
- 本地数据：`localStorage`
- 部署方式：静态站点部署，可使用 Vercel、Netlify、GitHub Pages、静态资源 CDN 或任意 Web 服务器

推荐原因：
- TypeScript 适合管理蛇身、色块、模式、分数等结构化游戏数据，能减少规则类 Bug。
- Vite 启动快、构建轻，适合小游戏快速迭代。
- React 适合承载首页、HUD、暂停、结算、设置、引导等界面层。
- Phaser 3 适合 2D 网格、输入控制、动画、音效和碰撞判定，能降低手写 Canvas 的成本。
- 纯前端版本可以最快验证玩法，不被登录、接口、数据库、部署链路拖慢。

### 14.2 前端分层建议
- React 负责页面框架、开始页、设置页、HUD、暂停弹窗、结算页、引导提示等非核心实时画面。
- Phaser 负责主游戏场景、蛇移动、色块生成、碰撞判定、消除动画、局内特效和音效触发。
- 游戏核心规则尽量放在可测试的 TypeScript 模块中，不要全部写死在 Phaser Scene 内，方便后续维护和单元测试。
- React 与 Phaser 之间通过轻量事件或状态回调通信，例如分数变化、游戏结束、暂停恢复、重开一局。

### 14.3 PC 与移动端输入
PC 端：
- 支持方向键。
- 支持 `WASD`。
- 禁止 180 度瞬间反向，避免蛇头直接撞回自身。

移动端：
- 支持滑动手势改变方向。
- 提供可选虚拟方向键，方便不习惯滑动的玩家。
- 触控区域应足够大，避免误触 UI 按钮。
- 页面应禁止游戏区域滚动和双击缩放干扰。

### 14.4 响应式布局要求
- 桌面端优先展示完整棋盘、右侧或顶部 HUD、操作说明和结算信息。
- 移动端优先保证棋盘最大化显示，HUD 精简放在顶部，虚拟按键放在底部。
- 棋盘尺寸应使用视口宽高动态计算，避免在小屏手机上溢出。
- 关键文字和按钮不得小于移动端可读可点的尺寸。
- 首版优先适配手机竖屏；横屏可保持可玩，但不作为首版重点打磨对象。

### 14.5 本地数据设计
当前版本只保存轻量本地数据：
- `pop_snake_best_score`：本机最高分
- `pop_snake_best_survival_seconds`：本机最长生存时间
- `pop_snake_best_combo`：本机最高 Combo
- `pop_snake_settings`：音效、震动、虚拟按键等设置
- `pop_snake_tutorial_completed`：是否完成过新手提示

注意：
- 本地数据不保证跨设备同步。
- 清除浏览器缓存后数据可能丢失。
- 本地最高分只能作为单机娱乐记录，不具备反作弊意义。

### 14.6 核心数据结构
- 蛇身建议使用有序数组或双端队列维护。
- 每节至少记录：网格坐标、颜色、节序。
- 色块记录：网格坐标、颜色、类型。
- 游戏状态记录：方向、下一方向、分数、Combo、存活时间、是否暂停、是否结束。
- 消除判定可基于蛇身颜色序列扫描连续段完成。

### 14.7 工程目录建议
- `src/app`：React 页面层，如首页、游戏页、结算页、设置页。
- `src/game`：Phaser 游戏核心逻辑。
- `src/game/scenes`：场景，如加载、主游戏、结算过场。
- `src/game/entities`：蛇、色块、特效等实体。
- `src/game/systems`：移动、碰撞、生成、消除、计分、输入等系统。
- `src/config`：颜色、数值、模式参数配置。
- `src/storage`：`localStorage` 读写封装。
- `src/assets`：图片、音效、字体资源。

### 14.8 关键技术难点
- 吃块后新增蛇节与移动逻辑的同步。
- 中间片段消除后，蛇身坐标如何重新衔接。
- 连锁消除的递归或循环判定。
- 色块生成的平衡性与无解保护。
- Phaser 画布在不同屏幕尺寸下的缩放与输入映射。
- 移动端触控、页面滚动、虚拟按键之间的冲突处理。

### 14.9 推荐实现顺序
1. 搭建 Vite + React + TypeScript 项目骨架。
2. 接入 Phaser，并让游戏画布在 PC 和移动端自适应显示。
3. 完成基础蛇移动、方向输入和碰撞失败。
4. 实现普通色块生成与吃块增长。
5. 实现颜色序列判定、消除和连锁消除。
6. 实现分数、Combo、结算、重新开始。
7. 接入 `localStorage` 保存本机最高分和设置。
8. 补充移动端滑动与虚拟方向键。
9. 打磨视觉、动效、音效和新手提示。
10. 最后再评估是否进入后端、排行榜、账号体系等下一阶段。

## 15. 后续后端扩展预留

### 15.1 扩展触发条件
只有当纯前端版本的核心玩法验证成立后，再考虑接入后端能力。

建议触发条件：
- 需要在线排行榜。
- 需要跨设备同步进度。
- 需要账号登录或玩家资料。
- 需要云端配置、活动、关卡下发。
- 需要更严肃的分数校验和反作弊。

### 15.2 后续可选方案
后续可继续使用以下方向：
- 数据与服务：Supabase
- 后端服务：Node.js + NestJS 或 Express
- 接口方式：REST API 或 Supabase Client / Edge Functions
- 鉴权方式：Supabase Auth，优先支持匿名登录，再逐步支持邮箱或第三方登录

### 15.3 后续数据表草案
如果进入后端阶段，可考虑以下表：
- `profiles`：玩家基础资料
- `run_records`：单局成绩记录
- `player_progress`：玩家进度与解锁状态
- `player_settings`：跨设备设置
- `game_configs`：云端游戏配置

### 15.4 当前阶段处理原则
- 当前开发中不创建 `server` 目录，除非明确进入后端阶段。
- 当前开发中不引入 Supabase SDK，避免增加环境变量和部署复杂度。
- 当前开发中不设计真实接口请求，所有配置从本地模块读取。
- 如需模拟未来接口，可使用前端 Mock 模块，但不得阻塞纯前端可玩版本。

## 16. 验收标准

### 16.1 核心玩法验收
- 玩家能在 1 局内自然理解“增长”和“消除”的关系
- 游戏不会频繁出现纯靠运气、无从决策的局面
- 三消成功时有明显正反馈
- 玩家可以通过策略控制长度，而非只能被动等死

### 16.2 MVP 完成标准
- 单局流程完整可玩
- 核心循环稳定，无明显逻辑错误
- 蛇尾三连消除、Combo、结算逻辑正确
- 平均单局时长控制在 1 到 3 分钟

## 17. 风险与注意事项

### 17.1 主要风险
- 如果消除太容易，游戏会失去贪吃蛇的紧张感
- 如果消除太难，玩家会觉得只是“更折磨的贪吃蛇”
- 如果色块刷新过于随机，策略空间会被运气吞没
- 如果动画太重，会破坏移动手感

### 17.2 设计把控建议
- 优先把核心循环打磨顺，再扩玩法
- 先验证“蛇尾三连同色消除”是否真的有趣，再决定要不要上复杂系统
- 特殊色块、Boss、养成都应建立在基础体验稳定之后

## 18. 结论
“消消蛇”的产品核心不是简单叠加两个经典玩法，而是围绕“颜色进入蛇身后形成可规划的消除链”建立新的生存循环。

首版最重要的不是功能多，而是把以下三个感受做出来：
- 吃块会紧张，因为会变长
- 凑消除会开心，因为能减负
- 每一步都值得想，因为颜色顺序会影响后续生死

如果这三个感受成立，这个项目就具备继续扩展关卡、特殊机制和长期成长系统的价值。
